# Skill: Analogy Builder

When called, map each technical component of the project to a real-world analogy.

## Process
For each key component in the project analysis:
1. Identify what the component DOES (its job, not its implementation)
2. Find a real-world equivalent that a non-developer would recognize
3. Explain the mapping — where the analogy holds and where it breaks down

## Output Format
Return a structured list:

**[Component Name]**
Real-world equivalent: [analogy]
Why it fits: [1 sentence]
Where the analogy breaks down: [1 sentence]

## Rules
- Analogies must be universally relatable (no domain-specific references)
- Never use "it's like a black box" or "it's basically a function"
- Prefer physical, spatial, or process analogies over abstract ones
- Each analogy should make someone say "oh, NOW I get it"

## Examples of strong analogies
- A message queue is like a deli counter ticket system — you take a number, wait your turn, nobody cuts the line
- An API is like a restaurant menu — you don't need to know how the kitchen works, just what you can order and what you'll get back
- A cache is like keeping your most-used tools on your desk instead of the storage room
